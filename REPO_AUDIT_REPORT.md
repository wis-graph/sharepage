# 📊 SharePage 프로젝트 코드 및 구조 검토 보고서

본 문서는 현재 `sharepage` 레포지토리(웹사이트 템플릿)의 코드 중복, 불필요한 처리, 그리고 구조적 최적화가 필요한 부분을 분석한 보고서입니다.

## 1. HTML 파일 중복 (index.html vs 404.html)
*   **현황**: 두 파일은 거의 99% 동일한 내용을 담고 있습니다. `404.html`은 GitHub Pages의 SPA 라우팅(Deep Linking)을 지원하기 위해 사용됩니다.
*   **문제점**: UI 구조나 라이브러리 버전(Katex, Highlight.js 등)을 변경할 때 두 파일을 모두 수정해야 하는 번거로움과 실수 가능성이 있습니다.
*   **의견**: 
    *   `index.html`을 소스로 유지하고, 빌드/배포 시점에 `404.html`을 복사 생성하는 방식으로 일원화하는 것이 좋습니다.

## 2. 정적 파일 중복 관리 (posts/ 디렉토리)
*   **현황**: `scripts/sync.js`에 의해 생성된 HTML 파일들이 직접 커밋되어 있습니다.
*   **문제점**: 마크다운 파일(`notes/`)이 수정될 때마다 `posts/` 폴더 내의 HTML 파일들도 매번 커밋/푸시해야 합니다. 이는 Git 히스토리를 지저분하게 만들고 충돌 위험을 높입니다.
*   **의견**: 
    *   `posts/` 폴더는 `.gitignore`에 추가하여 로컬에서는 생성만 하고, 실제 배포는 GitHub Actions(CI)에서만 수행하도록 변경하는 것을 검토해야 합니다.

## 3. CSS 로딩 방식의 비효율
*   **현황**: `package.json`에 `bundle.css` 빌드 스크립트가 있지만, 정작 `index.html`에서는 10개 이상의 개별 CSS 파일을 수동으로 불러오고 있습니다.
*   **문제점**: 불필요한 HTTP 요청이 발생하며, 파일 추가 시마다 HTML을 수정해야 합니다.
*   **의견**: 
    *   개발 시에는 개별 파일을 로드하되, 배포 시에는 `bundle.css` 하나만 로드하도록 최적화하거나, 현대적인 방식(Vite 등)의 도입을 고려할 수 있습니다.

## 4. 템플릿 로직의 이중화 (sync.js vs Client-side JS)
*   **현황**: `sync.js`(Node.js)는 SEO/OG 태그 삽입을 위해 HTML을 파싱/재조립하고, 실제 화면 렌더링은 `js/app.js`(Browser)가 담당합니다.
*   **문제점**: HTML 템플릿 구조가 변경되면 `index.html`은 물론 `sync.js`의 문자열 치환 로직도 함께 점검해야 할 수 있습니다. 
*   **의견**: 
    *   `sync.js`의 정규 표현식 기반 치환 방식을 좀 더 안정적인 템플릿 엔진 방식으로 교체하거나, HTML 내부에 주석 표기(e.g. `<!-- TITLE_SLOT -->`)를 활용해 치환 로직을 단순화할 수 있습니다.

## 5. 불필요한 테스트 및 작업 파일
*   **현황**: 루트에 `test.html`, `task.md` 등의 파일이 존재합니다.
*   **문제점**: 실제 서비스 배포 시 포함될 필요가 없는 파일들입니다.
*   **의견**: 
    *   개인적인 작업 노트나 테스트 파일은 별도의 `docs/` 또는 `temp/` 폴더로 옮기거나 삭제하여 배포 패키지를 깔끔하게 유지해야 합니다.

## 6. JavaScript 버전 파라미터 관리
*   **현황**: 캐시 방지를 위해 `?v=42000`과 같은 수동 파라미터를 사용 중입니다.
*   **문제점**: 코드를 수정할 때마다 수동으로 이 숫자를 올리는 것은 현실적으로 불가능합니다.
*   **의견**: 
    *   GitHub Action 배포 시점에 커밋 해시나 타임스탬프를 자동으로 삽입해주는 스크립트를 도입하여 자동화해야 합니다.
